Intuitive Management Interface for Graph Database

Filipe S. P. Prates · Federal University of Rio de Janeiro (UFRJ)

English version of the full content, with embedded figures and a reformatted code snippet for improved readability.

Abstract

This article describes an intuitive interface for managing graph data, allowing easy navigation and arbitrary manipulation of nodes and edges. As an example, it presents an implementation using a Neo4j database managed through a GraphQL API and a Vue.js web interface.

1. General Information

The available tools for managing data stored in graph-based databases, especially Neo4j Database, are usually limited to direct interactions through DCL (Data Control Language) and provide results as graphical views of subgraphs and tables.

When such data is managed by users with no prior knowledge of the schema, the results and visualizations may be difficult to understand, and there is also a risk of large-scale unintended changes to the data.

In this context, there arose a demand for an interface that would allow non-technical users to visualize and interact with data stored in a graph database.

2. The “Profiles and Lists” Interface

The developed interface, called “Profiles and Lists,” receives the schema of the associated database through the InformationSchema request. Using a profile page template and a node list page template, it allows arbitrary creation and editing of nodes and their relationships.

Each graph node then has a profile page that displays and allows editing of its properties and relations. In addition, each node label present in the database has a list/search page where users can search for and access the profile pages of nodes belonging to that label.

The advantage of this model is that every profile page and every list page follows the same logic, so the implementation code can be reused. In practice, the application consists of only one profile page and one list page, in addition to the login/authentication page. These are populated according to context, and actions are communicated through generic GraphQL requests that receive labels as arguments, allowing the same action to be reused in different contexts. This minimizes the application code size and makes maintenance easier.

2.1. Node Profile Page

The node profile page consists of the node’s properties at the top of the screen and, below that, a series of tabs, one for each different relationship label that the node may have.

Within each tab on the profile page there is a list of elements representing the node’s neighbors through relationships containing the label that identifies the tab. Users can select elements and, depending on their permissions, perform actions such as Add, Create, Edit, Delete, Remove Relationship, and any other action relevant to the node label. These actions are generated through GraphQL templates that receive the tab label and the owner node label as arguments, along with the identifiers of the selected list elements.

These element and neighbor lists also contain hyperlinks to the profile pages of the neighboring nodes. In this way, users can traverse the graph through relationships, each step taking place through node profile pages, with access to actions such as editing, creating, and deleting nodes, as well as creating and removing relationships between existing nodes.

2.2. Node List Page

However, the graph stored in the database may be disconnected, or the node the user wants to manage may be many steps away from the node currently displayed in the profile page, reachable through a non-obvious path, or it may be necessary to start graph navigation from some initial node. In these situations, one of the “List” pages and its search tools becomes useful.

These pages list all nodes of a given label in paginated form, with hyperlinks to their respective profile pages. They also provide generic search and filtering functionality through GraphQL queries that receive the list label as an argument, as well as direct actions on the listed elements.

3. Reusable Edit/Create Form

Inside the reusable form component, an object named properties is created and used to populate the form according to the needs of the label, as well as with the existing node information when the form is in edit mode.

Using the information returned by the introspection request, the editable properties are filtered and then separated according to their type. For each node property type, a specific interaction component is rendered in the form. This results in a set of text boxes, date selectors, boolean selectors, and even selectors with fixed values (enums), which are informed by the introspection request itself the first time the edit/create form for that label is opened.

Below is an example of rendering different form components according to their type. The snippet was restyled here only to improve readability, without changing the logical structure presented in the article.

Vue.js form example 
<form>
  <row
    v-for="(propInfo, propName) in properties.strings"
    :key="i"
  >
    <col full>
      <v-text-field
        v-model="propInfo.value"
        :label="propName"
        :disabled="propInfo.disableEdit"
        outlined
      />
    </col>
  </row>

  <row full
    v-for="(propInfo, propName) in properties.dateTimes"
    :key="j"
  >
    <col full>

        :label="propName"
      />
    </col>

</form>

To complete the interface, there is an upper state-management layer that manages forms and actions and populates the page templates through the information received from InformationSchema, the database introspection request from the neo4j-graphql library.

4. Results

The graph abstraction in “Profiles and Lists” proved to be an intuitive way to navigate and interact with the data, and it has been used daily since 2020 by multiple users from different areas of an education technology company to manage data stored in a Neo4j database.

5. Conclusions

A tool is needed to interact with data in an intuitive and simple way, thereby enabling several non-technical users to manage the same stored graph.

The “Profiles and Lists” interface simplifies interaction with a graph database by using interfaces and navigation patterns similar to social networks, with which users are already familiar.

The possibility of performing the same edge-manipulation action from either a node’s profile page or the profile page of its neighbor provides speed and convenience. The user can choose whichever path to the manipulation action is most convenient at the moment.
