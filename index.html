Abstract

Development of a data management interface for Neo4j graph databases through a GraphQL API, focused on intuitive visualization, navigation, and understanding of graph data, allowing arbitrary exploration and manipulation of nodes and relationships.

Keywords: Interface, Database, Graphs, GraphQL

1 Chapter 1 - Introduction
1.1 Motivation

Properly managing the data stored in a database is fundamental to any information system. That management must be both efficient and intuitive, so that incorrect changes are avoided and system users interact with updated and correct data.

Managing large amounts of data stored as graphs can be particularly complex. Available data-management tools, especially when dealing with Neo4j databases, are often limited to direct DCL interactions and result views such as graphical and textual representations, as in Neo4j Browser's graph result frame.

When the number of stored nodes and relationships becomes significant, the graphical or list-based view can be difficult to understand, especially for users who do not already know the data schema. In addition, there is a real risk of large-scale mistaken changes to the data.

1.2 Contribution

This work presents a data-management interface for Neo4j databases through a GraphQL API, with a focus on intuitive visualization, navigation, and management of the stored graph data. The interface receives the database schema and generates a "Profiles and Lists" environment that allows the creation and editing of nodes, their relationships, and each node's data.

At the time of writing, the system is used extensively by more than thirty collaborators in the company for which it was developed, especially by the Quality, Development, Support, and Educational teams.

The work documents both the user interface and the intermediate layer that connects the interface to the database. On the client side, the system is generated through reusable components and the Nuxt.js, Vue.js, and Apollo Client toolchain. On the server side, it uses Node.js, Express.js, and the official Neo4j library that connects the database to a GraphQL API, @neo4j/graphql.

The user interface follows a different navigation dynamic: it maps the isomorphism between a node and its neighbors into a profile page, its tabs, and its listed elements. In practice, that makes it possible to generate a profile page for every node in the database and to navigate through relations into other node profiles, while arbitrarily changing or adding data along the way.

1.3 Use of the interface in support of an adaptive learning and gamified assessment platform

The interface described in this work was initially developed at Jovens Gênios Provedor de Conteúdo LTDA, an education technology company. Jovens Gênios provides an adaptive learning and gamified assessment platform for students in public and private education systems across several Brazilian states.

Its main data domain consists of nodes representing school structures - school networks, schools, classes, students, teachers, and managers - as well as nodes representing the content created by the company - questions, summaries, topics, courses, and disciplines. It also includes nodes for platform activities such as assignments, diagnostic tests, battles, and especially nodes referring to students' answers, which connect each student to the question and to the activity that led the student to that content.

The tree structure of the topic organization, with questions in the leaf topics, the tree structure of the school organization, and the ease of maintenance and feature development with the tools available to the development team were determining factors in choosing a graph database, especially Neo4j.

As more content was produced and more students and classes were added to the graph, data management became an obstacle. Those entities change over time and need regular updates. To solve the issue of direct graph manipulation by users without deeper prior knowledge of the schema, the management interface described in this work was developed.

1.4 Roles of the interface users

The user profiles that use the interface in the company are:

  • Educational team: people responsible for direct contact with schools and for defining classes.
  • Support team: people who interact online with students and teachers to clarify questions, map issues, and solve possible problems.
  • Quality Control team: people who help set up tests and understand platform behavior in different scenarios.
  • Developers: people who need to inspect the state of the database graph and occasionally manipulate it.
2 Chapter 2 - Technologies Used
2.1 Three-layer architecture

The system follows a three-layer architecture in which a Vue.js interface sends requests over HTTP in GraphQL. On the server, each request generated by the user's interaction with the interface is translated into Cypher Query Language expressions, which are then executed in the database.

Figure 2.1. Three-layer architecture: Vue.js client -> GraphQL/HTTP -> Node.js and Express.js server -> Cypher -> Neo4j database.
2.2 Neo4j and Cypher Query Language

Neo4j is a graph-oriented database management system developed by Neo4j Inc. It operates on a data structure consisting of nodes, relationships, and properties. That makes Neo4j an ideal choice for scenarios in which relationships between pieces of data are as important as the data itself.

Each node and each edge carries one or more labels. Those labels instantiate a lookup index that behaves similarly to a table in a relational database, allowing efficient retrieval of all nodes or all edges of a certain label.

Neo4j does not use SQL as its data-control language. Instead, it uses its own language: Cypher Query Language.

Figure 2.2. Example of a Cypher query: find all nodes labeled Person with property name equal to "Dan", then follow a LOVES relationship and return the people loved by those Dan nodes.
2.3 GraphQL and @neo4j/graphql

The communication between interface and server uses a GraphQL API. GraphQL is a language for querying and manipulating data that allows a single server endpoint to listen to and resolve different requests from different client interfaces. A request is specified in GraphQL syntax, usually in .gql files, and sent in the body of an HTTP POST request.

The server exposes a series of queries and mutation requests. In all operations it also allows request configuration through optional arguments and by letting the client choose exactly which properties of the response type it wants to receive.

The client can therefore invoke any available function and choose exactly the subset of the response needed at that moment. Because GraphQL has a tree structure that resembles JSON, it is possible to traverse edges in the requested object graph and return precisely what the frontend needs, reducing bandwidth and avoiding unnecessary payload.

To generate all CRUD queries and mutations for the node and edge types available in the database, the project uses the official Neo4j library that also provides the language for defining and generating the schema: @neo4j/graphql.

GraphQL query example.gql
query GetSchoolStudentsWithAssignmentGrades($schoolId: ID!) {
  schools(where: { id: $schoolId }) {
    students {
      id
      name
      assignments {
        id
        grade
      }
    }
  }
}

If the client also needs the assignment description, it can simply add that field to the request. The same is true for any property or relationship defined on the type.

Extended field selection.gql
assignments {
  id
  grade
  description
}
GraphQL mutation example.gql
mutation ChangeStudentEmail($id: ID!, $newEmail: String) {
  updateStudent(
    where: { id: $id },
    update: { email: $newEmail }
  ) {
    student {
      id
    }
  }
}

Those requests are resolved in a server that uses @neo4j/graphql together with type definitions such as the following:

Simplified schema definitiontypes.graphql
type School {
  id: ID
  students: [Student] @relationship
}

type Student {
  id: ID
  assignments: [Assignment] @relationship
}

type Assignment {
  id: ID
  grade: Float
  description: String
}
2.4 Apollo

Apollo is a JavaScript state-management library built around GraphQL. It is composed of two parts that communicate with each other.

  • Apollo Client: used in the client application. It provides data-request control, loading state handling, cache management, and local state variables, simplifying GraphQL requests.
  • Apollo Server: a GraphQL server compatible with any client that sends it GraphQL requests, including Apollo Clients. It is used to generate the schema that requests and returned data must follow, and it can connect to any managed data source - in this project, a Neo4j database.
2.5 Node.js, 2.6 Express.js, and 2.7 Vue.js

Node.js is an open-source platform based on the V8 engine that allows JavaScript code execution on multiple platforms.

Express.js is the most popular web framework for Node.js. It is minimalist and provides the basic web-development capabilities needed for this project, including HTTP verb handling and middleware layers for authentication and authorization.

Vue.js is a framework for building web interfaces. A .vue file contains three parts:

  • Template: HTML structure that organizes components on screen and interacts directly with JavaScript via directives such as v-for, v-if, and v-model.
  • Script: logic for the component, including database requests and interface-state control.
  • Style: CSS rules that define component appearance.
3 Chapter 3 - Description of the Developed System

The guiding objective of the interface was to provide a friendly way to manage data in a complex Neo4j-backed graph database. The system needed to allow a user to understand the graph's current state and to create, edit, connect, and disconnect arbitrary nodes stored in the database.

3.1 Client application

The client application's goal is to be the interface with which a user lacking prior knowledge of the stored data can interact. For every action performed in the client, a GraphQL request is generated and sent in an HTTP POST request to the server's GraphQL endpoint.

The pages described below were developed with Vue.js.

3.1.1 Profiles and Lists

Together with the Design team, the project identified a way to understand and visualize the graph stored in the database by giving each node a profile page that shows and allows editing of its properties and relations, and by giving each relevant label in the database a list/search page. From that isomorphism, it became possible to create an interface that allows users to navigate and edit the graph intuitively.

Figure 3.1. Abstraction in which an owner node is related to nodes of several labels and that neighborhood is represented as a profile page with tabs and lists.

There are therefore two main screen types:

  • Profiles: the owner node's properties appear at the top of the page, while tabs appear below - one tab for each type of relationship that the owner node has. Inside each tab, there is a list of neighboring nodes connected by that relationship type. Each element has hyperlinks to the corresponding neighbor profile pages. This makes it possible to traverse the graph through nodes and relations while repeatedly passing through the current node's profile and performing actions such as editing, creating, or connecting nodes.
  • Lists: navigation through the graph needs a starting node. For that reason, list pages display all elements of a given type, in paginated form, with hyperlinks to their profile pages. They also provide generic search and filtering functions and direct actions on listed elements.
Figure 3.2. Educator profile page. The blue and gray actions in the neighbor-list header include editing, removing, and creating edges between the profile node and selected elements, plus tab-specific actions such as freezing or unfreezing a class.

The advantage of this modeling is that every profile page and every list page follows the same rule. As a result, the code that implements them can be reused. In practice, the application consists of a single profile page and a single list page, plus the login/authentication page, together with maps that define each case's specific behavior.

That decision makes maintenance easier, reduces code size, and therefore also tends to reduce production bugs. The frontend is composed of the two main system pages, an upper layer that controls global forms and actions, and several specific Vue components used by both pages. It also includes JSON files describing what is specific to each profile and each type, such as which special actions are allowed depending on the profile type and the active tab type.

Figure 3.3. Directory organization and communication between parts of the "Profiles and Lists" interface codebase developed in Vue.js for Jovens Gênios.
3.1.2 Use of IntrospectionQuery

IntrospectionQuery is a request provided by @neo4j/graphql that asks the server for the stored data schema and for the queries and mutations allowed by the API. This makes it possible to know all possible requests, their arguments, and their types, as well as all data types as they are defined in the type definitions.

Those data are used, for example, when an edit form is opened. The response is used to dynamically assemble the number and type of input components according to the properties reported for the type. If a new property is added to the backend type definition, the frontend will automatically show a field for it without requiring a new frontend deployment.

This dynamic construction of forms and profiles was useful and was employed for several months. Over time, however, it became clear that not every attribute of every node should be directly editable by users of the management system. Some fields should be filled automatically by other flows rather than edited manually, and showing every property in the admin interface ended up confusing users.

The project therefore adopted a hybrid approach: the frontend still receives the attributes of the relevant type when building create and edit forms, but filters them so that only editable properties are shown. Those filters are defined through maps stored in the utils directory. The same hybrid strategy is used for list-table columns and for the properties displayed in the profile header.

The search and filtering component in list pages also benefits from IntrospectionQuery, because it can automatically list all possible values of a given ENUM property in a selector. Knowing the type of each property makes it possible to render the appropriate input component for user interaction.

3.1.3 Reusable form

Inside the form component, an object named properties is populated with the information returned by IntrospectionQuery and filtered according to the rules configured in utils, which determine which properties are editable. For the remaining ones, the interface renders a specific interaction component depending on the property's type.

3.1.4 Actions

The data-manipulation actions exposed to the user through the interface are divided into three categories:

  • Profile actions: available on every node profile page and applied directly to the owner node. They include deletion, property editing, and optionally other custom actions.
  • Tab actions: available in the header of each tab list and visually represented with a darker color. They refer to the owner node's relationships with nodes of the selected tab type. They include creating a new node of the tab's type already connected to the owner, connecting the owner to an existing node of that type, and optionally other custom actions. They do not depend on selecting list elements.
  • Element actions: available in the same list header but shown with a lighter color. They operate on selected list elements and include editing a selected node, disconnecting it from the owner, and optionally other custom actions. They do depend on current selection.
Figure 3.5. Profile action interface for a node of type Student.

Every action, regardless of category, is propagated to the Control layer. That layer receives an object - whose structure varies according to action category - containing the information that defines the user's intended operation.

In the Control layer it becomes possible to choose which stored .gql request file should be sent to the server, or to build the request dynamically.

Deleting multiple selected studentsGraphQL + JavaScript
// graphql/Student/delete.gql
mutation DeleteStudent($Id: ID!) {
  deleteStudents(where: { id: $Id }) {
    nodesDeleted
  }
}

// control.js
async handleDelete(payload) {
  for (let el of payload.elements) {
    if (confirm(`Are you sure you want to delete ${el.name}?`)) {
      try {
        await this.$apollo.mutate({
          mutation: require(`@/graphql/${payload.type}/delete.gql`),
          variables: el,
        });
      } catch (e) {
        console.error(e);
        alert(e);
        return;
      }
    }
  }
}
3.2 Server with GraphQL endpoint

The interface does not communicate directly with the database. The server acts as an intermediate layer that authenticates the request, generates the resulting Cypher, and sends an execution request to the database.

3.2.1 Type definitions

One of the server's main responsibilities is to define every type that exists in the database. At this stage - analogous to defining tables and fields in a relational database - the domain data are modeled and every possible property of each entity that can be manipulated through the management system is defined.

Type definitions are extensive and split into files with well-defined scopes. Their primary function is to define properties, property types, and possible relationships. They are also where custom queries and mutations are declared when they go beyond the CRUD operations automatically generated by neo4j-graphql, including logic that may involve communication with other systems.

The library uses those type definitions to generate CRUD operations, whose execution manages nodes in the database with labels matching the type names.

Simplified type-definition fileGraphQL schema
""" Battles types """
type Battle implements Activity @node(labels: ["Battle", "Activity"]) {
  id: ID @id
  status: ActivityStatus
  createdAt: DateTime! @timestamp(operations: [CREATE])
  updatedAt: DateTime! @timestamp(operations: [CREATE, UPDATE])
  topic: Topic @relationship(type: "SELECTED_IN_BATTLE", direction: OUT)
  task: Task @relationship(type: "BATTLE_IN_TASK", direction: OUT)
}

enum ActivityStatus {
  NOT_STARTED
  IN_PROGRESS
  DONE
}

type Mutation {
  mergeStudentsBattle(studentsIds: [ID!], battleId: ID!, type: ContextType): [ID]
  setBattleResult(battleId: ID!): String
  deleteBattle(battleId: ID!): ID
}
3.2.2 Resolvers

Some properties defined in the schema are not necessarily stored in Neo4j itself. For example, a property returning the ranking of an activity may actually come from a Redis database in another server. In those cases, the system needs a way to redirect requests and define an alternative logical flow responsible for returning the property's content.

Resolvers are functions used when the default direct connection flow through neo4j-graphql needs to be changed, extended, or even prevented. They can fetch data from other databases, execute specific Cypher statements, trigger update processes, or implement any other required logic.

3.2.3 Connection to the database

The project uses the neo4j-driver library, which authenticates with username and password and then creates a driver object, establishing a direct connection between the server and a Neo4j instance.

Neo4j driver initializationJavaScript
import neo4j from "neo4j-driver";

const driver = neo4j.driver(
  process.env.NEO4J_URI || "bolt://localhost:7687",
  neo4j.auth.basic(
    process.env.NEO4J_USER || "neo4j",
    process.env.NEO4J_PASSWORD || "neo4j"
  ),
  {
    maxConnectionLifetime: 8 * 60 * 1000,
    maxConnectionPoolSize: 50,
    connectionAcquisitionTimeout: 2 * 60 * 1000,
    disableLosslessIntegers: true,
  }
);
3.2.4 Authentication and authorization

After defining resolvers and type definitions, security becomes essential. The system must prevent any connected user from executing any mutation exposed by the endpoint.

For authentication, the project uses Neo4jGraphQLAuthJWTPlugin, which validates requests through a token passed in the Authorization header of every GraphQL request after login. The token carries, in encrypted form, the identifier of the user who generated it, that user's type - such as Student, Teacher, or Admin - and a signature that allows its authenticity to be checked.

After the token is validated, an authorization layer is still needed for subsequent requests to ensure that the user is allowed to execute the specific GraphQL operation. A student, for example, must not be allowed to execute a mutation that deletes a school.

The @neo4j/graphql stack provides Role-Based Access Control features, making it possible to define which operations - CREATE, READ, UPDATE, DELETE - are allowed for which user types and for which data types, while also using the user identifier contained in the token for finer-grained rules.

Those rules are critical for system security because schema information is exposed to clients through IntrospectionQuery and GraphQL permits generic data modification.

Request authorization helperJavaScript
export function isRequestAllowed(req) {
  const token = req.headers?.authorization?.trim().split(" ")[1];

  if (parseJwt(token)?.userType === "ExternalAdmin") {
    const operationName = req.body?.query
      .match(/(?<=\{)\s*[a-zA-Z0-9]*/)?.[0]
      .trim();

    return EXTERNAL_ADMIN_ALLOWED_REQUESTS.includes(operationName);
  }

  return true;
}
3.2.5 Schema generation

The schema for the Neo4j database is generated after instantiating a Neo4jGraphQL() object from @neo4j/graphql. It receives the connected driver, type definitions, resolvers, and any plugins used by the project.

Generating the schemaJavaScript
import { Neo4jGraphQL } from "@neo4j/graphql";
import { Neo4jGraphQLAuthJWTPlugin } from "@neo4j/graphql-plugin-auth";
import { resolvers } from "./resolvers";
import { typeDefs } from "./typeDefinitions";

async function initializeNeo4jGraphQL() {
  const neo4jGraphQL = new Neo4jGraphQL({
    driver,
    typeDefs: gql`\${typeDefs}`,
    resolvers,
    plugins: {
      auth: new Neo4jGraphQLAuthJWTPlugin({
        secret: process.env.JWT_SECRET,
        algorithms: ["HS256"],
        credentialsRequired: false,
      }),
      config: {
        enableDebug: process.env.NODE_APP_ENV === "staging",
      },
    },
  });
  return await neo4jGraphQL.getSchema();
}

Then Apollo Server is initialized with that schema, and Express is used to expose it on the configured routes and ports.

Publishing the endpointJavaScript
import { expressMiddleware } from "@apollo/server/express4";

const app = express();
const apolloServer = new ApolloServer({
  schema: schema,
  introspection: true,
  plugins: [
    // ...
  ],
});

app.use(
  path,
  expressMiddleware(apolloServer, {
    context: ({ req }) => ({
      driver,
      neo4jDatabase: process.env.NEO4J_DATABASE,
      req
    })
  })
);
4 Chapter 4 - Conclusions
4.1 Results

The system developed and described in this work proved to be a useful and efficient data-management interface and has been used daily since 2020 by multiple users from different areas of the company to manage data stored in a Neo4j database.

The abstraction of the graph into "Profiles and Lists" proved to be an intuitive way to navigate and interact with the data. The possibility of performing the same action through different paths provides speed and flexibility in data-manipulation tasks.

In the Support area, for example, only a two-hour training session during onboarding is enough for new workers to understand and use the system. Even without prior knowledge of the data model, users can begin using the system as a work tool, under supervision, on the very same day they first encounter the interface.

4.2 Future work

Possible future developments include:

  • Automatically generating the .gql request files from the information returned by IntrospectionQuery. Constantly configuring type definitions was a point of difficulty for the development and maintenance of the system.
  • Developing a server-agnostic version of the "Profiles and Lists" interface for graph-data management: an interface capable of connecting to any GraphQL endpoint with open IntrospectionQuery and, after authentication, allowing arbitrary visualization and modification requests over the data.
